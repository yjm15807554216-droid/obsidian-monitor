# 监控系统架构说明

> 返回首页：[[00_告警追踪系统_首页]]

## 项目代码结构

```
~/crypto_monitor/
├── venv/                    # Python 虚拟环境
├── config.py                # 配置文件（API Key, 阈值参数）
├── symbol_mapping.py        # 币种符号映射表
├── auto_mapping.py          # 自动生成的扩展映射表
├── data_sources.py          # 数据获取（币安 + CoinGecko）
├── analytics.py             # 分析模块（变化率、分位数）
├── alerting.py              # 告警逻辑和 Telegram 推送
├── database.py              # SQLite 历史数据存储
├── monitor.py               # 主程序
├── build_mapping.py         # 自动构建映射表的工具
└── monitor.db               # SQLite 数据库文件
```

## 数据流

```
[币安 API]                       [CoinGecko API]
   |                                  |
   | (OI, 价格, 24h 数据)              | (流通市值)
   |                                  |
   v                                  v
[data_sources.py] <----------------- 缓存 15 分钟
   |
   v
[monitor.py] 主循环（每 60 秒）
   |
   |---> 计算 OI/MC 比率
   |
   |---> [database.py] 写入 monitor_data 表
   |
   |---> [analytics.py] 计算变化率和分位数
   |
   |---> [alerting.py] 判断是否触发告警
                |
                v
        [Telegram Bot] 推送消息
```

## 三类告警的触发逻辑

### 1. 绝对值告警（V2）

```
if 控筹率 ≥ 25%: 严重级 🔴
elif 控筹率 ≥ 20%: 警告级 🟠
elif 控筹率 ≥ 15%: 观察级 🟡
```

**去重机制：** 同一个币只在等级提升时再次告警。

### 2. 变化率告警（V3）

```
if 1小时内变化 ≥ +5 百分点: ⚡ 急速飙升告警
if 4小时内变化 ≥ +8 百分点: ⚡ 急速飙升告警
```

**冷却时间：** 同一个币 1 小时内不重复告警。

### 3. 历史分位数告警（V3）

```
if 当前值在过去 14 天数据的分位数 ≥ 95%:
    🚨 创新高告警
```

**前置条件：** 数据库中至少有 20 个该币的历史记录。

## 监控范围

- **范围**：24h 成交量排名前 50 的 USDT 永续合约
- **过滤**：
  - 排除超大市值币（>$50 亿）—— 无控盘可能
  - 排除黑名单（XAU/XAG/USDC 等贵金属和稳定币）
  - 排除已下架/停盘合约

## 数据源说明

| 数据 | 来源 | 频率 | 成本 |
|------|------|------|------|
| 合约 OI | 币安 API | 每分钟 | 免费 |
| 现货价格 | 币安 API | 每分钟 | 免费 |
| 24h 涨跌 | 币安 API | 每分钟 | 免费 |
| 流通市值 | CoinGecko API | 缓存 15 分钟 | 免费 |

**币安 API 限制：** 2400 请求/分钟（监控 50 个币每分钟约 100 请求，远低于限制）

**CoinGecko API 限制：** 免费档约 10-30 请求/分钟（用 15 分钟缓存规避）

## 部署环境

**当前：** 本地 MacBook Pro

**计划迁移：** 海外云服务器（Vultr 东京 / 阿里云日本节点）

**迁移原因：**
- Mac 关机/休眠会导致数据断档
- 数据断档影响历史分位数计算的准确性
- 长期 24/7 运行需要稳定环境
